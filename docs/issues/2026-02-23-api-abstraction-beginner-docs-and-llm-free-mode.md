﻿# API 추상화·초보자 문서·LLM-Free 운영 모드 이슈 정리 (2026-02-23)

## 1. 배경

- Aetherius는 하나의 코드베이스에서 CLI, HTTP 서버 API, 브라우저 콘솔을 모두 제공하는 하이브리드 엔진이다.
- 현재도 README, ARCHITECTURE, CLI_USAGE, LLM 이슈 문서 등으로 개념·구조가 잘 설명되어 있지만,
  - (1) API 관점에서는 여전히 `/api/command` + 문자열 명령 중심의 저수준 인터페이스에 가깝고,
  - (2) 초보자가 “어디서 무엇을 먼저 해야 하는지”를 빠르게 파악하기에는 문서가 깊고 길며,
  - (3) LLM/Science Orchestrator가 잘 분리되어 있음에도, “LLM 없이도 완전히 운영 가능한 모드”가 계약 수준에서 명시되어 있지는 않다.
- 이 이슈는 위 세 가지 축을 한 번에 다루어,
  - **도메인 고수준 API 레이어 설계**
  - **초심자 친화 문서 구조 재정비**
  - **LLM-Free 운영 모드 명시 및 모듈 경계 정리**
  를 아키텍처 수준에서 명시하는 것을 목표로 한다.

## 2. 현 구조 정리

### 2.1 API 및 실행 모드

- 진입점: `src/main.ts`
  - `--mode cli` → 로컬 터미널 기반 CLI 클라이언트 (`src/app/cli.ts`)
  - `--mode server` → Express 기반 HTTP 서버 (`src/app/server/server.ts`, `router.ts`)
- 서버 모드 REST 엔드포인트(예시):
  - `/api/status`, `/api/tick`, `/api/command`, `/api/science`, `/api/narrative`,
    `/api/snapshots`, `/api/events`, `/api/dataset`, `/api/linear-model`, `/api/species` 등
- 특징:
  - 대부분의 제어는 여전히 문자열 기반 명령(`/api/command { "cmd": "..." }`)에 의존한다.
  - CLI 명령 세트와 HTTP API는 개념적으로 연결되어 있지만, **역할/시나리오 단위의 고수준 API 레이어**로 추상화되어 있지는 않다.

### 2.2 문서 구조

- 핵심 문서:
  - [README.md](../README.md): 프로젝트 비전, 아키텍처 개요, 실행 방법, 기본 API 예시.
  - [ARCHITECTURE.md](../ARCHITECTURE.md): 레이어/디렉터리 구조, 확장/리팩터 포인트.
  - [CLI_USAGE.md](../CLI_USAGE.md): CLI 명령 목록, 역할(신/탐험가/과학자/운영자) 모드, 예시 루프.
  - LLM/Science 관련: `docs/issues/2026-02-20-gemini-llm-orchestration.md` 등.
- 장점:
  - 엔진 수준 아키텍처와 CLI 설계 철학이 상세하게 정리되어 있다.
  - 역할 기반 게임 모드, 실험 루프 예시 등 고급 사용자를 위한 내용이 충실하다.
- 한계(초보자 관점):
  - “15분 안에 한 번 돌려본다” 수준의 **아주 얕은 Quickstart**가 분리되어 있지 않다.
  - CLI/HTTP/Web 중 어디서 시작해야 하는지, 어떤 루프를 따라가야 하는지가 흩어져 있다.
  - LLM이 없는 환경에서 어떤 기능이 가능한지, 어떤 기능이 빠지는지에 대한 **요약된 가이드**가 없다.

### 2.3 LLM 및 Science Orchestrator

- 관련 파일:
  - `src/ai/llmService.ts` — Gemini / OpenRouter 등 LLM 호출 서비스.
  - `src/ai/orchestrator.ts` — ScienceOrchestrator, `mode: 'full' | 'lite'`, `maxAgents`, `enablePeerReview`, `enableRebuttal` 등 옵션.
  - `docs/issues/2026-02-20-gemini-llm-orchestration.md` — SDK/엔드포인트/쿼터 이슈와 개선 방향 정리.
- 현재 상태:
  - LLM 호출 계층은 코드 레벨에서 잘 분리되어 있고, `lite` 모드 등으로 오케스트레이션 무게를 조절할 수 있다.
  - 하지만 “엔진 자체는 LLM 없이도 완전히 운영 가능하다”는 사실이, **운영 계약이나 모드 개념으로 명시되어 있지는 않다.**
  - LLM 장애/쿼터 이슈 시의 기대 행동(어디까지 degrade 되는지)도 문서상에서 명확히 드러나지 않는다.

## 3. 요구사항 정리

### 3.1 API 추상화 요구사항

1. **도메인 시나리오 중심 고수준 API 레이어**
   - `/api/command` + 문자열 명령 레이어 위에,
     - “환경 시나리오 조성”, “생명/개체 실험”, “장기 진화 관측” 같은 **도메인 시나리오 단위의 HTTP/CLI API**를 정의한다.
   - 각 시나리오 API는:
     - 초기 조건, 진행 방식, 완료 조건을 인자로 받아
     - 내부적으로 여러 개의 저수준 명령/틱 진행을 조합해 실행한다.

2. **역할 기반 API·명령 그룹**
   - 신(System), 탐험가(Explorer), 과학자(Scientist), 운영자(Operator) 역할별로:
     - 해당 역할이 자주 수행하는 행위를 **고수준 엔드포인트/명령 그룹**으로 묶는다.
   - 예:
     - 신: 시간 제어, 환경 개입, 재해/축복
     - 탐험가: 지도 조회, 장소 탐색, 개체 관찰
     - 과학자: 실험 시나리오 정의, 데이터 수집, 분석 요청
     - 운영자: 상태 모니터링, 스냅샷/이벤트 전송, 백업/복구

3. **저수준 명령 호환성 유지**
   - 기존 CLI 명령 및 `/api/command`는:
     - 고급 사용자/디버깅/스クリپ트 용도로 계속 유지한다.
   - 고수준 API는 내부에서 저수준 명령을 사용하더라도,
     - 외부 계약은 **도메인 객체와 시나리오**를 기준으로 정의한다.

### 3.2 초보자 문서 요구사항

4. **15분 Quickstart 루프 제공**
   - “완전 초보자” 대상으로:
     - 설치 → 실행 → 세계 상태 관찰 → 간단한 개입 → 결과 관찰까지 이어지는 **단일 짧은 루프**를 문서로 제공한다.
   - CLI 기반, HTTP + Web 콘솔 기반 중 하나를 선택할 수 있도록 둘 다 간단 예제를 준비한다.

5. **역할별 Quickstart 섹션**
   - 각 역할별로:
     - 필수 개념 3~5개
     - 대표 명령/엔드포인트 3~5개
     - 전형적인 루프 예시 1~2개
     를 한 페이지 이내로 요약한 섹션을 만든다.

6. **용어집·다이어그램 정리**
   - Tick, World, Place, Entity, EventBus, Recipe 등 핵심 용어를:
     - 비전공자도 이해할 수 있는 비유/간단한 설명으로 재정리한다.
   - 간단한 다이어그램(텍스트 기반 또는 이미지)을 통해 데이터 흐름을 보여준다.

7. **LLM 의존/비의존 영역 명시**
   - 문서 전반에서:
     - “LLM 불필요” 기능과 “LLM 있을 때 추가로 가능한 기능”을 시각적으로 구분한다.
   - 예: 아이콘, 태그, 주석 등을 사용.

### 3.3 LLM-Free 운영 요구사항

8. **LLM-Free 모드 정의**
   - 엔진은 환경 변수 또는 설정 파일을 통해:
     - “LLM-Free 모드”를 명시적으로 선택할 수 있어야 한다.
   - 이 모드에서는:
     - LLM 관련 기능(API/CLI 명령)는 “비활성” 또는 “제한된 placeholder 응답”을 제공하되,
     - 코어 시뮬레이션·관측·실험 기능은 모두 정상 작동해야 한다.

9. **모듈 경계와 헬스 체크**
   - 운영자 관점에서 다음 모듈을 독립적으로 이해/운영할 수 있어야 한다.
     - 시뮬레이션 코어 (World, Tick, Environment, Entities)
     - 데이터 계층 (스냅샷/이벤트/통계)
     - CLI/HTTP/Web 콘솔 레이어
     - LLM/Science Orchestrator 레이어
   - 각 모듈은:
     - “켜기/끄기” 옵션과
     - 헬스 체크/상태 조회 방법이 문서와 API로 정의되어야 한다.

10. **LLM 장애 시 graceful degradation**
    - LLM API 에러(404, 429, 네트워크 장애 등)가 발생해도:
      - 엔진 전체가 중단되지 않고,
      - LLM 관련 기능만 제한되며,
      - 상태 API/로그를 통해 “AI 보조 기능이 제한된 상태”를 확인할 수 있어야 한다.

11. **사람이 사용할 수 있는 운영 플레イブ북**
    - LLM 없이도:
      - 세계 초기화, 실험 설정, 모니터링, 백업/복구까지의 **운영 플로우**를 사람이 따라 할 수 있어야 한다.
    - 이 플레イブ북은:
      - 명령/엔드포인트가 아니라 “행동 단계(1~N)” 중심으로 서술한다.

## 4. 비기능 요구사항

1. **학습 곡선**
   - 초보자는 15분 이내에:
     - 최소 한 번의 “개입 → 결과 관찰” 루프를 경험할 수 있어야 한다.
2. **일관성**
   - CLI, HTTP API, Web 콘솔, 문서에서 사용하는 용어·역할·시나리오 이름은 일관되어야 한다.
3. **관찰 가능성(Observability)**
   - LLM의 활성/비활성 상태 및 장애 여부는:
     - 상태 API/로그/콘솔에서 쉽게 확인 가능해야 한다.
4. **확장성**
   - 향후 새로운 도메인(문명/사회/경제 등)을 추가해도:
     - 같은 패턴으로 고수준 API/문서/운영 모드를 확장할 수 있어야 한다.
5. **모듈 독립성**
   - LLM 모듈을 제거하거나 교체해도:
     - 코어 엔진의 API/CLI/운영 문서 계약은 깨지지 않아야 한다.
